# ТЗ: Инструмент резервирования и миграции КЭП (КриптоПро)

**Рабочее имя:** `CertMigrator` (заменить при необходимости)
**Назначение:** объединить в одном UI две операции над ключевыми контейнерами КриптоПро — **копирование контейнера** и **снятие признака неэкспортируемости** — и решить главную боль: сделать выбор контейнера человекочитаемым (владелец, организация, срок действия), а не по обезличенному GUID.

Целевой сценарий: **резервирование и миграция КЭП** сотрудников (десятки контейнеров), а не разовое копирование.

---

## 1. Границы возможного (читать до реализации)

Инструмент работает **только с файловыми контейнерами** (флоппи-профиль КриптоПро), в том числе лежащими на токене в режиме FKC. Это не ограничение реализации, а граница физики:

| Носитель / режим | Копирование | Снятие флага |
|---|---|---|
| Реестр | да | да |
| Файловый контейнер на токене (Рутокен Lite/S, ЭЦП 2.0 в режиме FKC) | да | да |
| Ключ, сгенерированный **внутри** чипа (Рутокен ЭЦП 2.0 / JaCarta-2 ГОСТ, крипта на токене) | **невозможно** | н/п |

**Требование:** аппаратно-сгенерированные ключи (режим «крипта внутри токена») должны определяться и **исключаться из списка копируемых** заранее, чтобы оператор не тыкался вслепую. Признак — невозможность экспорта на уровне провайдера + отсутствие извлекаемого ключевого материала.

Зависимость от **КриптоПро CSP неустранима** — сам ГОСТ-провайдер не реализуется. Инструмент это проверяет, но не заменяет.

## 2. Функциональные требования

### 2.1 Инвентаризация контейнеров с резолвом сертификата (ядро)
Собрать список всех доступных контейнеров и для каждого разрешить связь `контейнер → сертификат`, показав в таблице:

- **Владелец** (Subject CN)
- **Организация** (Subject O)
- **Действует до** (NotAfter) — с подсветкой истекающих
- **Носитель** (реестр / имя токена из FQCN)
- **Флаг exportable** (виден заранее)
- **Тип ключа** (файловый / аппаратный — для исключения аппаратных из копирования)
- **Thumbprint** (SHA1) — технический идентификатор

Это то, чего нет в штатных инструментах: они показывают GUID-имя контейнера (`he-8f3a1c2d.000`) и заставляют определять сертификат по косвенным признакам.

### 2.2 Копирование контейнера
Носитель → носитель или носитель → реестр. Поддержка выбора целевого носителя. Для массового сценария — «скопировать выбранные на носитель X».

### 2.3 Снятие признака неэкспортируемости
Пересборка контейнера: чтение приватного ключа, гашение бита неэкспортируемости, запись обратно. Только для файловых контейнеров.

### 2.4 Перепривязка сертификата (обязательный шаг миграции)
После копирования на новый носитель сертификат в личном хранилище остаётся привязан к **старому** контейнеру. Инструмент должен уметь «скопировать + перепривязать» — переустановить сертификат с привязкой к новому контейнеру, иначе система продолжит искать старый носитель. **Не опциональная фича — часть операции миграции.**

### 2.5 Массовые операции
Чекбоксы в таблице, групповое действие над выборкой. Фильтр/сортировка по «Действует до» и по организации.

## 3. Архитектура

Слои разделены так, чтобы резолвер и ядро операций не зависели от UI и были переиспользуемы.

```
[UI: таблица контейнеров + групповые действия]
        │
[Проверка окружения]
   ├── CSP установлен? (версия 4.0 / 5.0)
   └── перечисление носителей / токенов
        │
[Резолвер: контейнер ↔ сертификат]   ← самая мясная и переиспользуемая часть
   ├── путь 1: сертификат внутри контейнера → читать напрямую
   └── путь 2: матч по хранилищу My (CRYPT_KEY_PROV_INFO → pwszContainerName)
        │
[Ядро операций над контейнером]
   ├── keycopy (носитель → носитель / реестр)
   ├── снять флаг exportable (пересборка header.key)
   └── перепривязка сертификата к новому контейнеру
        │
[КриптоПро CSP]   ← неустранимая зависимость
```

## 4. Резолвер (реализовать первым)

**Принцип: работать через CryptoAPI, а не через парсинг stdout.**

Основной путь — перечисление хранилища и чтение свойства провайдера:

- `CertOpenStore` / `CertEnumCertificatesInStore` — обход `uMy`.
- `CertGetCertificateContextProperty(CERT_KEY_PROV_INFO_PROP_ID)` → `CRYPT_KEY_PROV_INFO` → поле `pwszContainerName` даёт мостик `сертификат → контейнер`.
- Из контекста сертификата — Subject (CN, O), NotAfter, Serial, SHA1 Thumbprint.
- Флаг exportable / тип ключа — через провайдер (`CryptAcquireContext` + попытка/свойства ключа).

**Фолбэк (CLI)** — держать, но не как основу:
- `csptest -keyset -enum_cont -verifyc -fqcn` — список контейнеров с полным именем (виден носитель).
- `certmgr -list -store uMy` — поле `Container` связывает сертификат с контейнером.

> Важно: формат вывода `csptest` / `certmgr` и написание флагов **плавает между версиями CSP 4.0 и 5.0**. Парсинг текста в проде — источник поломок при обновлении CSP. CLI только фолбэк, основа — CryptoAPI.

## 5. Технологический стек (предложение, обсуждаемо)

Требуется прямой доступ к CryptoAPI/CAPICOM под Windows:

- **C++ / C#** — естественная интеграция с CryptoAPI (`crypt32.dll`, `advapi32.dll`). C# через P/Invoke даёт быстрый UI (WinForms/WPF) при полном доступе к API.
- Установщик — **Inno Setup** или **WiX**: кладёт бинарь, проверяет наличие CSP, при отсутствии уводит на страницу загрузки.
- Целевая ОС — Windows (КриптоПро CSP).

Решение по языку — за реализатором; критерий: минимум прослоек между кодом и CryptoAPI.

## 6. Безопасность и журналирование

Инструмент снятия неэкспортируемости в чужих руках — механизм кражи подписи. Раз он собирается в удобный UI, встроить сдерживающие механизмы **в ядро**, а не опционально:

- **Журнал операций**: что скопировано/пересобрано, когда, с какого носителя на какой, Thumbprint. Локальный, неизменяемый (append-only).
- **Подтверждение перед деструктивными/чувствительными действиями** (снятие флага, перезапись контейнера).
- Никакого автоматического экспорта «всё подряд» без явного выбора оператора.

Цель — не формальное соответствие, а чтобы собственный инструмент не вернулся инцидентом.

## 7. Порядок реализации (этапы)

1. **Резолвер** (раздел 4) как отдельный модуль + консольная проверка: на выходе таблица `Container ↔ Thumbprint ↔ Subject(CN,O) ↔ NotAfter ↔ exportable`. Это ядро ценности, тестируется независимо от UI.
2. **Определение типа ключа** — отделить файловые контейнеры от аппаратно-сгенерированных, пометить последние как некопируемые.
3. **Операция копирования** (keycopy) поверх резолвера.
4. **Перепривязка сертификата** к новому контейнеру.
5. **Снятие флага exportable**.
6. **UI**: таблица с колонками из 2.1, чекбоксы, групповые действия, подсветка истекающих.
7. **Журналирование** (раздел 6).
8. **Установщик** с проверкой CSP.

## 8. Открытые вопросы для проработки в Claude Code

- Формат хранения журнала (файл / встроенная БД).
- Нужен ли экспорт таблицы инвентаризации (CSV) для аудита парка КЭП.
- Поведение при копировании контейнера, у которого сертификат отсутствует и в контейнере, и в хранилище (обрабатывать как «orphan», показывать отдельно).
- Стратегия для CSP 4.0 vs 5.0: единый CryptoAPI-путь vs ветвление.
