# Запись на токен и копирование токен → токен (проектная заметка)

Статус: обсуждение, не реализовано. Резюме анализа.

## Ключевое различие: «тупое» хранилище vs «умный» токен

| | Папка / реестр / HDIMAGE | Токен (Рутокен) |
|---|---|---|
| Как хранит | просто 6 файлов | файлы + on-token ACL, доступ стережёт чип |
| Запись | сырой `WriteFile` — работает | сырой записи мало: нужны правильные ACL |

Чтение с токена сырьём (rtComLite `ReadBinary`) безопасно — ACL не трогаем.
Запись сырьём небезопасна: без правильных ACL на `primary.key`/`masks.key`
либо чип отвергнет контейнер, либо приватный ключ станет доступен без PIN.
CryptoPro при создании контейнера выставляет эти ACL, аллоцирует ID папки,
привязывает к структуре носителя. Повторить сырьём = реверсить логику
создания контейнера Рутокена.

## Почему CryptoAPI-импорт не спасёт

`CryptImportKey(PRIVATEKEYBLOB)` требует, чтобы ключ был экспортируемым, а весь
парк неэкспортируемый. Инъекция ключа через API отпадает.

## keycopy НЕ подходит для неэкспортируемых ключей (проверено)

Прежнее предположение «keycopy копирует зашифрованный блоб как есть» —
**неверно**. Проверка на стенде (два Рутокена Lite, дефолтный PIN):

```
csptest -keycopy -contsrc "\\.\Aktiv Rutoken lite 1\<cont>" \
        -contdest "\\.\Aktiv Rutoken lite 0\<new>" -pinsrc 12345678 -pindest 12345678
-> CopyPrivateKey!ExportKey(PRIVATEKEYBLOB, NULL): Error 0x8009000b (NTE_BAD_KEY_STATE)
```

`csptest -keycopy` внутри вызывает `CryptExportKey(PRIVATEKEYBLOB)` — а он
падает на **неэкспортируемом** ключе (весь наш парк такой). То же с
`-typesrc/-typedest`. Приёмник при этом не создаётся.

Вывод: **токен → токен для неэкспортируемых контейнеров через csptest keycopy
невозможен.** В папку/реестр/HDIMAGE копирование работает (сырьё, без экспорта);
на токен — нет.

## Что остаётся для записи на токен

- **Сырьё через rtComLite** — прочитать 6 файлов с источника и записать на
  токен-приёмник (WriteBinary + структура + ACL). Это то, чем, вероятно,
  копируют токен→токен утилиты без экспорта. Риск: правильные ACL на файлы
  ключа; без них чип отвергнет или ключ станет доступен без PIN. Требует
  вскрытия того, как Рутокен раскладывает контейнер. Только на своём стенде,
  на истёкшем контейнере.
- **Штатная копия CryptoPro** (cpconfig → «Скопировать контейнер») копирует
  неэкспортируемые контейнеры файловым способом (не экспортом), в т.ч. на
  токен — но это GUI, не скриптуется через csptest.

Итог: простого CLI-примитива для записи неэкспортируемого контейнера на токен
нет. Нужен либо реверс сырой записи через rtComLite (риск), либо интеграция с
внутренним механизмом CryptoPro (не публичный API).

## Механика, о которой помнить

1. PIN обеих сторон (дефолт подставляем сами, иначе спрашиваем; журнал по ТЗ 6).
2. Свободная память приёмника (`FreeMemory` через rtComLite) — Рутокен Lite
   64K тесный.
3. Уникальное имя контейнера на приёмнике; дружественное имя из `name.key`.
4. Аппаратные контейнеры (крипта в чипе) записать так же нельзя — блоба нет.
5. Проверка после записи: резолвер + `csptest -check`.

## Что проверить на стенде до реализации

- Точный синтаксис `keycopy` токен→токен и флаги PIN.
- Работает ли keycopy токен→токен одной командой или нужен промежуточный шаг.
- Поведение с дефолтным PIN без GUI (`-silent`?).
